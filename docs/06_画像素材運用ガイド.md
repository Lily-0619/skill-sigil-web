## 2. スキル画像

### フォルダ

```text
画像/スキル画像/{クラスコード}/{スキル名}.png
```

現状は上記（スキル名のみ）で格納されている。

### ファイル名規則（2026-07-13決定）

管理の都合上、次の規則へリネームする。

```text
画像/スキル画像/{クラスコード}/{クラスコード}_{no}_{スキル名}.png
```

- 例: `画像/スキル画像/WR/WR_01_ハートレスブロウ.png`
- `no` はスキルの表示順（`04_データ設計書.md` の `skills.display_no`）を使う。特別スキル1〜4・通常スキル1〜13の通し番号か、`special`/`normal`を区別する番号かは、リネーム実行時に対応表とあわせて確定する。
- 対応表案を作成し、まなの承認後に一括リネームを実行する（実行前に対象パスと変更内容を提示する）。

### ルール

- Excelのクラスコードとフォルダ名を一致させる。
- スキル名はマスタの日本語名と一致させる。
- ファイル名に使えない文字は置換ルールを決める。
- 見つからない場合は共通の「画像なし」アイコンを表示する。
- 同名スキルが存在する場合は内部IDをファイル名へ加える。

## 3. クラスボタン

```text
画像/class_button/{クラスコード}.png
```

現在、Excelの30クラスに対して31枚あり、`UC.png` が追加で存在する。

2026-07-13決定: UCは現時点でスキルデータを持たないため、クラス選択画面には表示しない（`classes.enabled=false`）。将来クラスが正式追加されデータが揃った時点で表示対象へ切り替える。

また、クラス選択画面ではクラスボタンの下に略称（例: WR）を表示し、クラス名はゲーム内クラス名をカタカナ音写した名称で表示する（例: WR＝ウォーリア）。将来クラスが増えた場合も同じ規則で追加できるようにする。

## 4. キャラクター背景

### 現在の配置

```text
画像/character/*.png
```

### 表示仕様

- クラス選択画面: 全画像からランダム
- クラス選択後: 選択クラスに対応する画像からランダム
- 直前と同じ画像は、候補が2枚以上なら避ける
- ページを開いた時だけ決め、入力中に勝手に切り替えない
- 背景透明度の目安は8〜14%
- 人物は右下または左下へ寄せる
- UIの背面へ配置し、クリックを受け取らない

## 5. 追加を簡単にする仕組み

画像一覧をソースコードへ手入力しない。起動・ビルド時に素材フォルダを走査し、画像マニフェストを自動生成する。

追加作業:

1. 対応フォルダへPNGを入れる。
2. 開発環境の素材同期またはビルドを実行する。
3. 自動検証結果を確認する。
4. 公開サイトを再公開する。

公開済みサイトへローカル画像を置くだけでは反映されないため、再公開は必要。

## 6. クラス判定

キャラクター画像は、マニフェストに `class_id` と `variant` を持たせる。

例:

```text
class_id: WR
variant: classic
path: character/m_img_warrior_classic.png
```

既存ファイル名が英語クラス名のため、クラスコードとの対応表を1つだけ管理する。新しい画像は既存命名規則に合わせるか、対応表へ追加する。

### 命名規則（2026-07-13決定）

新規追加分は `{クラスコード}_{連番}.png`（例: `WR_01.png`）で統一する。2026-07-13時点の実測では、一部（約十数枚）が新形式（`AC_1.png`など、ゼロ埋めなし）へ移行済みで、残り大半は旧形式（`m_img_◯◯_awaken.png`等）のまま。旧形式の一括リネーム可否・連番のゼロ埋め統一は、キャラクター背景表示の実装計画時に判断する。

## 7. 自動検証

- 読み込める画像形式か
- 0バイトではないか
- クラスコードが存在するか
- スキル名とマスタが一致するか
- 同じパスが重複していないか
- 背景候補が選択クラスに1枚以上あるか
- 推奨サイズを大きく外れていないか

エラー画像は公開候補から除外し、一覧で報告する。

## 8. 軽量化

- 元画像を保管したまま、公開用には必要サイズへ変換する。
- PNGの透明度が必要な素材はPNGまたはWebPを使う。
- 初期画面では必要な背景1枚とクラスボタンだけを読み込む。
- スキル画像は選択クラス分だけ読み込む。
