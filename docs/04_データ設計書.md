# スキル秘伝HP データ設計書

## 1. 設計原則

- ゲーム共通マスタと、ユーザー固有データを分離する。
- Excelのセル位置をそのままアプリ内部構造にしない。
- IDは表示名から独立させ、名称変更でデータが壊れないようにする。
- 現在の17スキルを表示するが、件数を固定値として実装しない。
- 不明な数値は文字列で保持し、後から編集できる。
- ユーザーが削除を確定するまでデータを失わない。

## 2. データの区分

### マスタデータ

- クラス
- スキル
- 固定秘伝枠
- 秘伝タイプ
- 効果候補
- 画像対応

### ユーザーデータ

- 所持秘伝
- 保存編成
- 装着状況
- 設定
- バックアップ履歴

### effect_master

| 項目 | 型 | 必須 | 説明 |
| :---- | :---- | :---- | :---- |
| effect_id | string | 必須 | 変更しない内部ID |
| sigil_type_id | string | 必須 | 対応する秘伝タイプ |
| name_ja | string | 必須 | 効果名 |
| value_hint | string | 任意 | `%`、秒など入力のヒント |
| value_abyssal_a | string | 任意 | 深淵の既定値1（`-`は該当等級なし） |
| value_abyssal_b | string | 任意 | 深淵の既定値2（2値効果のみ） |
| value_primal_a | string | 任意 | 太古の既定値1 |
| value_primal_b | string | 任意 | 太古の既定値2（2値効果のみ） |
| value_chaos_a | string | 任意 | 混沌の既定値1 |
| value_chaos_b | string | 任意 | 混沌の既定値2（2値効果のみ） |
| sort_order | number | 必須 | 表示順 |
| enabled | boolean | 必須 | 現行候補か |

効果ごとの実数値は `11_効果マスタ数値.md` を正本とする。`A / B` 表記は「A か B」（同一等級内で取りうる2値＝下位/上位）で、`value_*_a`（下位）/ `value_*_b`（上位）に保持する。無欠の `X , Y%` だけは別で、2種類の効果の同時付与（`value_*_a`＝X、`value_*_b`＝Y として扱う）。等級に `-` がある効果はその等級の秘伝が存在しない（煌めくは混沌のみ）。数値を持たない効果（例: 即時発動化）は全 `value_*` を空にし、効果の有無だけを扱う。系列タイプは深淵のみ存在（10%統一）、太古・混沌の秘伝は存在しない（2026-07-13確定、`value_primal_*` / `value_chaos_*` は空欄固定）。

## 4. 装着判定

装着可能条件はすべて満たす必要がある。

1. スキルが `sigil_eligible=true`。
2. 対象の `slot_no` が存在する。
3. 固定枠の `sigil_type_id` と所持秘伝のタイプが一致する。
4. 所持秘伝の残数が1以上。
5. 効果が無効化された旧マスタの場合は警告し、新規装着を禁止する。

残数:

```text
available = inventory.quantity - count(equipped_sigils in current build)
```

複数編成は同時使用ではないため、MVPでは編成ごとに所持数制約を判定する。

## 5. 数値の扱い

効果数値のマスタが判明したため（`11_効果マスタ数値.md`）、MVPは **マスタ既定値＋手編集可能** とする。所持秘伝で `effect_id` と `rarity`（深淵/太古/混沌）を選ぶと、対応する既定値が自動で入る。ユーザーは `value_text` で上書き編集できる（ゲーム更新・個体差に対応）。

`value_text` の手編集例:

- `5%`
- `1.5秒`
- `+1回`
- `攻撃力+20 / 防御力+15`

空欄を許可する。マスタに既定値がある効果でも、ユーザーが空欄化・上書きできる。

## 6. マスタ更新と互換性

- `schema_version`: アプリの保存形式変更時に更新
- `master_version`: ゲームのスキル・効果変更時に更新
- マスタ更新で名称が変わってもIDは維持する。
- 削除されたスキル・効果は即消去せず `enabled=false` とする。
- 古い編成は読み取り可能な状態を維持し、再編集時に修正を促す。
- 現在の受け入れ条件は17スキルだが、データ構造は18以上にも対応する。

## 7. ローカル保存

IndexedDBに、マスタの参照版とユーザーデータを保存する。画像本体は保存せずURLだけを参照する。

保存単位:

- 所持品の追加・変更・削除
- 装着・解除・移動
- 編成の追加・変更・削除
- 設定変更

## 8. 画像マニフェスト

実装時に画像フォルダを走査し、次の一覧を自動生成する。

- `class_images`: class_id / variant / path
- `skill_images`: class_id / skill_name / path
- `class_buttons`: class_id / path

追加画像はビルド時に自動反映する。公開サイトへ反映するには再ビルド・再公開が必要。

### 命名規則（2026-07-13決定、詳細は`06_画像素材運用ガイド.md`）

- `画像/character/`: `略称_連番`（例: `WR_01.png`）。略称部分を`classes.code`と突合してclass_idへ対応付ける。
- `画像/スキル画像/<クラス略称>/`: `class_no_スキル名`（例: `WR_01_ハートレスブロウ.png`）。旧ファイル名（スキル名のみ）からのリネームを予定（`08_未決事項・リスク一覧.md` R-002関連）。
- `画像/class_button/`: `略称.png`（例: `WR.png`、既存のまま）。
