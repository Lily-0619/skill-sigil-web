# 16-3 技術説明 人間向け

この文書は、なぜ編成画面のコードがぐちゃぐちゃになりやすいのか、今後AIへどう頼むと事故を防げるかの説明です。

## いま何が危ないのか

`BuildEdit.tsx` は、画面表示だけでなく、スキル選択、装着枠選択、秘伝候補、所持秘伝、Free秘伝、確認モーダル、細かい条件分岐まで同じファイルに集まっています。
そのため、少し仕様を変えるだけでも別の場所に影響しやすい状態です。

たとえると、1つの大きな机の上に、説明書、部品、工具、完成品、メモを全部広げている状態です。
作業はできますが、新しい部品を足すたびに何かを落としやすくなります。

## ぐちゃぐちゃになりやすい理由

### 1. 状態と見た目が近すぎる

Reactでは「どのスキルを選んでいるか」の状態と、「そのスキルをどう表示するか」の見た目を分けると安全です。
今はそれらが近くにありすぎるため、表示を少し直すつもりが選択条件まで触ってしまう危険があります。

### 2. 似たUIが増えている

スキル一覧、所持秘伝、Free秘伝、効果一覧は、どれも「候補を並べて選ぶ」UIです。
似ているけれど条件が違うため、同じファイルにあると読み間違いが起きます。

### 3. 条件分岐が増えている

直近では、等級・数値選択と即時装着の条件を細かく修正しています。
こういう条件は1つ増えるだけなら簡単ですが、積み重なると「どの条件が優先か」が分かりにくくなります。

## なぜ分割するのか

分割の目的は、見た目を豪華にすることではありません。
将来まなが修正地獄にならないようにするためです。

分割すると、たとえば「装着確認モーダルだけ直したい」ときに、そのファイルだけ見れば済みます。
また、Claudeなど別AIに頼むときも「この部品だけ直して」と指示しやすくなります。

## 今後AIへ頼むときの言い方

よい頼み方:

```text
BuildEdit.tsxの見た目は変えず、装着確認モーダルだけを
src/components/build-edit/EquipConfirmModal.tsx に切り出してください。
ロジック変更はしないでください。
切り出し後に npm run build を実行してください。
```

避けたい頼み方:

```text
編成画面をいい感じに整理して、ついでに使いやすくして。
```

後者は、分割・デザイン変更・仕様変更が混ざりやすく、バグの原因になります。

## 共有画像生成機能の技術方針

将来的に「一覧画像として保存」を作る場合、画面をそのままスクリーンショットするより、共有専用カードを作る方が安全です。

理由:

- ボタンや余計な余白が写りにくい。
- X、Discord、スマホ保存など用途別サイズを作りやすい。
- 背景画像やフォント崩れに対応しやすい。

共有カードに入れる候補:

- クラス名
- ビルド名
- スキル構成
- 各スキルの秘伝
- パッシブ秘伝
- 作成日
- サイト名

サイズ候補:

- 横長: X・Discord向け
- 正方形: SNS共有向け
- 縦長: スマホ保存向け
- 背景あり・透過の切り替え

## セット比較機能の技術方針

セット比較は、編成管理と相性がよい機能です。
最初から全部を細かく比べるより、差分が分かる画面を目指します。

比較画面案:

```text
セットA                 セットB
スキル構成              スキル構成
秘伝効果                秘伝効果
共通部分                共通部分
異なる部分              異なる部分
効果の合計              効果の合計
```

便利にする工夫:

- 同じ秘伝は薄く表示する。
- 違う秘伝だけ色をつける。
- セットAにしかない効果を表示する。
- セットBにしかない効果を表示する。
- 数値効果の合計を比較する。

## ミニスライド: コード整理の考え方

### Slide 1: 今の状態

- 1ファイルに役割が集まりすぎている。
- 直す場所が分かりにくい。
- 小さな修正が別機能へ影響しやすい。

### Slide 2: 目指す状態

- パッシブ説明、スキル一覧、装着枠などを部品ごとに分ける。
- 見た目を直すファイルと、装着ルールを直すファイルを分ける。
- 1回の修正範囲を小さくする。

### Slide 3: AIへの指示のコツ

- 「見た目を変えない」「ロジックを変えない」を明記する。
- 1回に1部品だけ頼む。
- 実行する確認コマンドも指定する。
- 変更したファイル名を必ず報告してもらう。

### Slide 4: これからの進め方

- 第2回で分割を実施する。
- 第3回前に公開方針を詰める。
- 第4回で初回操作ガイドやOGPを追加する。
- 第5回でスマホ表示を強化する。
