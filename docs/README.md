# docs 作業ログと設計方針の入口

この `docs/` は、これまでの計画・設計・判断理由・今後の修正方針を残す場所です。
人間もAIも、実装作業に入る前にまずここを確認します。

## 読み方のルール

1. まずこの `docs/README.md` を読む。
2. 次に、今回の作業に関係する番号のMDを読む。
3. タイトルに「(完了)」が付いているMDは、原則として実装済みの記録です。ユーザーから明示指示がない限り、優先して読み返さなくてよいです。
4. タイトルに「方針」「未完」「次回」などがあるMDは、今後の実装前に確認します。
5. 実装が完了したら、該当MDのタイトルに「(完了)」を付け、必要なら結果と注意点を追記します。

## 16系ドキュメント

- `16-1-roadmap-for-humans.md`: 人間向けの今後の方針・優先順位・数回に分ける作業計画。
- `16-2-coding-instructions-for-ai.md`: AI同士が引き継ぐための具体的な実装指示・分割方針・事故防止ルール。
- `16-3-technical-notes-for-owner.md`: 人間向けの技術説明。なぜコードがぐちゃぐちゃになりやすいか、今後どう指示すれば防げるかを説明します。

## 今後の記録方針

- 大きな仕様変更は、コード変更の前にdocsへ意図を残します。
- 「見た目の変更」か「保守性のための整理」かを明記します。
- Claudeや別AIが作業する可能性を考え、必要ならサンプルコード・分割後のファイル名・禁止事項も書きます。
- WindowsとMacの両方で動かす前提で、OS依存のコマンドやパスは避けます。
